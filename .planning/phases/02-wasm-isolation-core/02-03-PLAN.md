<objective>
Deliver the instance pool with lazy instantiation and session lifecycle management. After this plan, a developer can acquire a session from the pool (getting a Store with host functions, fuel, and epoch configured), execute Wasm code, and release the session (Store dropped, memory reclaimed). The pool enforces concurrency limits and tracks active sessions. A stress test proves 1,000 concurrent sessions stay within the 64MB cap.

Purpose: Instance pool management is the final piece of the Wasm isolation vertical — it wraps the Engine, Linker, and SessionState infrastructure from 02-01/02-02 into a concurrency-safe pool that the agent runtime (Phase 3) can use directly. The benchmark (D-06) validates the 5ms P99 cold start target before considering optimization.

Output: InstancePool with acquire/release API, Semaphore-based concurrency bounding, DashMap session tracking, benchmark latency measurement, and stress test proving 1,000 concurrent sessions fit within 64MB each.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Session A -> Session B | Store-per-session isolation. SessionHandle drop destroys Store. Zero data residue. |
| Pool acquire flow | SessionState injected into new Store. Capabilities set at creation, never modified. |
| Pool -> OS Resources | Semaphore bounds max concurrent sessions. DashMap provides O(1) session lookup. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-13 | Information Disclosure | SessionHandle Drop | mitigate | Store dropped on SessionHandle drop. PoolingAllocator zeros memory slots before reuse. Session isolation test verifies no data leakage between sessions per SEC-01. |
| T-02-14 | Denial of Service | Semaphore | mitigate | max_concurrent bounds prevent unbounded session creation. acquire() blocks (async backpressure) when pool exhausted. Configurable at pool creation. |
| T-02-15 | Denial of Service | PoolingAllocator | mitigate | Stress test verifies 1000 concurrent sessions at 64MB each. PoolingAllocatorConfig max_memory_size=64MB not 4GiB default. Failure mode: acquire fails with clear error, not OOM kill. |
| T-02-16 | Tampering | InstancePre | mitigate | InstancePre is Arc-shared and read-only. No mutable state shared across sessions. Each acquire creates a new Store with fresh SessionState. |
</threat_model>
