- Define LoopConfig struct (derives Clone) with: max_iterations: u32, model: String. Default: max_iterations=20, model="gpt-4o".
- Define MockLlm trait: #[async_trait] pub trait LlmProvider: Send + Sync { async fn chat(&self, messages: &[String]) -> anyhow::Result<String>; }. This is a mockable interface for testing; Plan 02 replaces with async-openai.
- Define pub async fn react_loop(config: &LoopConfig, session: &mut SessionHandle, llm: &dyn LlmProvider, tx: &mpsc::Sender<ReActStep>) -> anyhow::Result<Vec<ReActStep>>:
  - Init trace: Vec<ReActStep> and history: Vec<String> from request context
  - For turn in 0..config.max_iterations:
    - session.store_mut().set_fuel(1_000_000).map_err(...)?  (per D-10 Pitfall 3)
    - Call llm.chat(&history).await? to get response text
    - Emit Thought via tx.send, push to trace
    - Emit Finished via tx.send, push to trace, break
    - Error handling: if any step fails, emit Error variant and return error
  - If loop exits without break (all turns used): return anyhow error describing MaxIterationsReached
  - Return Ok(trace)

## Trust Boundaries

| AgentRequest.user_message -> Agent Loop | Untrusted natural language input crosses into the agent execution path. Prompt injection risk. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | AgentRequest.user_message | accept | Prompt injection via user_message is a known limitation. Documented in RESEARCH.md security domain. Full mitigation in Phase 6 with guest-controlled guard prompts. MVP: system prompt instructs model to distinguish user input from instructions. |
| T-03-02 | Denial of Service | ReAct loop infinite execution | mitigate | D-08 tokio::select! with max_iterations (default 20) + wall_clock_timeout (default 5 min). Iteration limit checked inside loop body; wall-clock timeout raced via tokio::select!. Phase 2 Wasm fuel limits per turn provide inner-layer protection. |
| T-03-03 | Denial of Service | Wall-clock timeout bypass | mitigate | The run_with_guard function in guard.rs uses tokio::select! to race the loop future against tokio::time::sleep. The sleep future fires unconditionally; no code path can reset or extend the timeout. T-03-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages from untrusted registries. anyhow 1.0 is from crates.io (standard Rust error handling crate, ~10M+ downloads). All other deps already in lockfile. |
</threat_model>
