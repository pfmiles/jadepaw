## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Build system (Cargo) | Crate dependencies from crates.io — external code enters the build pipeline |
| Developer filesystem | `.gitignore` blocks `.env` secrets from version control |
| CI pipeline | GitHub Actions runner executes crate build scripts — supply chain trust boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | crates.io dependency supply chain | mitigate | `deny.toml` bans openssl/openssl-sys (forces rustls), `[bans]` denies multiple-versions and wildcards, `[advisories]` denies yanked crates. cargo-audit scheduled workflow in Plan 01-02 covers RustSec advisories. |
| T-01-02 | Information Disclosure | Version control (.gitignore) | mitigate | `.gitignore` blocks `.env` files; `.env.example` explicitly allowed. `.DS_Store` and IDE dirs excluded. |
| T-01-03 | Spoofing | crates.io typosquatting | accept | Verified via `cargo search --registry crates-io` in RESEARCH.md Package Legitimacy Audit. All packages confirmed as legitimate crates.io entries. Risk accepted at the research level — ongoing verification via cargo-audit in CI. |
| T-01-04 | Elevation of Privilege | Crate build scripts (build.rs) | accept | Rust crate build scripts run during compilation with filesystem access. All crates in this phase are Bytecode Alliance/tokio-team maintained (high trust). No custom build.rs written in Phase 1. Ongoing monitoring via cargo-deny advisories. |
| T-01-SC | Tampering | npm/pip/cargo installs (dev tools) | mitigate | cargo-nextest, cargo-deny, cargo-audit are Rust crates installed via cargo install from crates.io. Package Legitimacy Audit in RESEARCH.md confirms crate identity. cargo-deny itself validates all transitive dependencies. |

## Security Posture Summary

Phase 1 (infrastructure scaffold) has minimal attack surface — no runtime code, no user input, no data access. Security measures focus on supply chain integrity: `deny.toml` with license allow-list and ban on `openssl`/`openssl-sys`, `.gitignore` secrets exclusion, and `rustls` preference over `openssl`. ASVS Level 1 applies at the supply chain level only (V10 Malicious Code Protection, V14 Configuration).
</threat_model>
