## Phase Goal

**As a** developer using jadepaw, **I want to** register external tools (file I/O, HTTP) through a typed Tool abstraction and dispatch them from the agent loop, **so that** the agent can take real actions beyond pure reasoning.

## MVP Slice 3: ReAct Loop Integration (wire ToolRegistry, replace placeholder, MCP interface)

This plan completes the tool system vertical slice by wiring the `ToolRegistry` (Plan 01) and tool implementations (Plan 02) into the ReAct execution loop. After this plan, the agent's `LlmDirective::Act { tool, args }` branch dispatches through the registry instead of emitting a placeholder observation. Tool results are formatted as `ReActStep::Observation` with `is_error` discrimination, appended to the LLM message history so the agent can adapt. The system prompt is augmented with available tool descriptions per MCP `tools/list` format. The SSE stream is updated to include `is_error` in observation events.

    ///
    /// Tools are injected in MCP tools/list format so the LLM knows what

    pub fn build_system_prompt_with_tools(
        base_prompt: &str,
        tools: &[jadepaw_core::ToolDefinition],
    ) -> String {

            return base_prompt.to_string();

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM output → react_loop() | `LlmDirective::Act { tool, args }` — tool name and args from LLM are untrusted input. Validated by ToolRegistry::call_tool() (name lookup → UNKNOWN_TOOL if missing, capability check → CAPABILITY_DENIED if not whitelisted). |
| ToolResult → SSE stream | Observation strings may contain arbitrary content from tool output. The SSE event builder (axum) handles newline encoding. JSON serialization via serde_json prevents injection. |
| ToolResult → LLM message history | Tool output is appended to messages as a user message. Large tool outputs (>1MB) are already capped by HttpRequestTool per Plan 02. Observation strings are capped at 50KB in to_observation_string() per RESEARCH.md Pitfall 2. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-10 | Spoofing | Tool name injection in LLM output | mitigate | LLM can only invoke tools registered in ToolRegistry. Unknown tool names → UNKNOWN_TOOL error (retryable=false). The LLM cannot register new tools or call nonexistent ones. |
| T-04-11 | Information Disclosure | Tool output in SSE stream | accept | SSE observation events carry tool output. The SSE stream is authenticated at the transport layer (axum). Tool output may contain sensitive data — this is a feature, not a bug (the user should see what their tools return). |
| T-04-12 | Denial of Service | Large observation strings bloating LLM context | mitigate | Two layers of protection: (1) HttpRequestTool caps response body at 1MB (Plan 02). (2) `ToolResult::to_observation_string()` should truncate output to 50KB for LLM consumption per RESEARCH.md Pitfall 2. The full result is in `ToolResult::Ok { data }` for programmatic access. |
| T-04-13 | Elevation of Privilege | Bypassing ToolRegistry via direct tool.call() | mitigate | `react_loop()` only calls `tool_registry.call_tool()` — never `tool.call()` directly. Code review enforcement: `grep "\.call(args" crates/jadepaw-agent/src/loop.rs` must return 0 matches (only `call_tool` should appear). |
| T-04-SC | Tampering | cargo installs for this plan | accept | No new dependencies. All types used (Tool, ToolResult, ToolRegistry) are from Phase 01/02 already in the dependency tree. |
</threat_model>
